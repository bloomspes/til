# til

하루하루 배운 것과 통찰을 장기로 쌓는 개인 지식베이스. 한 번 쓰고 버릴 메모장이 아니라, 5년 뒤에도 자기 자신을 다시 읽게 만드는 흔적.

## 왜 쌓는가

개발자의 하루는 수많은 글·도구·패턴을 스쳐 지나간다. 그 순간엔 "다시 봐야지" 싶지만, 탭이 닫히는 동시에 대부분 사라진다. 1년이 지나면 무엇을 배웠는지 흐릿하고, 3년이 지나면 "내가 저걸 알고 있었던 것 같은데"라는 감각만 남는다.

이 레포는 그 흐릿함에 저항하는 실험이다. 단순 북마크 모음은 아니다 — 북마크는 링크일 뿐 기억이 아니다. 여기엔 매일 흘러 들어오는 raw 트렌드, 월말마다 그것들을 훑어 자기 언어로 다시 쓴 요약, 외부 글에서 뽑아낸 번역 자료가 쌓인다. 시간이 누적되면 raw 피드는 잊혀도 괜찮다. 살아남는 건 내가 직접 쓴 문장이다.

철학적으로는 간단하다. **읽은 것은 흘러가고, 쓴 것은 남는다.** Claude Code와 Codex는 수집과 초안을 맡지만, 증류하고 결론 내리는 자리는 사람이 지킨다. AI가 점점 더 많은 걸 대신해 주는 시대일수록 "내가 직접 쓴 내 언어"는 희소해진다. 그걸 쌓는 공간.

## 구조

```
til/
├── sandbox.md             # 매일 흘러 들어오는 raw 피드 (읽고 지나감)
├── summaries/             # 월말 증류 — 한 달을 돌아본 내 문장 (남는 것)
├── notes/                 # evergreen — 번역, 깊은 노트
├── archive/               # 과거의 흔적 (참고용, 수정하지 않음)
├── skills/                # Claude Code 스킬 (automation)
└── update-sandbox.sh      # 일일 수집 스크립트
```

세 가지 속도가 공존한다. **빠른 층(`sandbox.md`)**은 매일 갱신, **중간 층(`summaries/`)**은 월 1회 응축, **느린 층(`notes/`, `archive/`)**은 필요할 때만 건드린다. 모든 정보를 같은 주기로 다루려 하면 결국 어느 하나도 유지되지 못한다는 경험에서 나온 구조다.

파일링 규약과 커밋 규칙은 [AGENTS.md](AGENTS.md)를 참고.

## 리듬

- **매일** — `./update-sandbox.sh` 한 번. codex가 최근 48시간 Hacker News·GitHub Trending을 훑어 `sandbox.md`에 append. 읽거나 흘려보낸다. 가벼움이 핵심.
- **월말** — (Phase 2) `/synthesize-month`로 그 달의 `sandbox.md` 섹션을 훑어 반복 주제와 흐름을 초안으로 제안. 사용자가 자기 언어로 다시 써서 `summaries/YYYY-MM.md`를 확정. 이 지점에서 진짜 지식이 태어난다.
- **수시로** — 읽은 외부 글은 `analyze-article` 스킬로 번역해 `notes/translations/`에. 한 주제를 깊게 쓰고 싶을 땐 `notes/` 아래 직접 마크다운.
- **연말** — 12개 `summaries/` 파일을 훑어 "이 해의 관심사"를 한두 단락으로 정리. 개인 기술 연감이 된다.

## 방향

- **Phase 1 — 구조 정리 (완료, 2026-04).** 2년간 방치된 실습 디렉토리를 묻고, raw / summary / notes / archive 네 축으로 루트를 재편. 지금 그 상태.
- **Phase 2 — 합성 루프 (다음).** `/synthesize-month` 스킬. 반복 등장한 주제를 codex가 뽑고, 사용자가 골라 자기 문장으로 다시 쓴다. 자동 요약만 생성해 두진 않는다 — 그건 또 다른 raw 층에 불과하니까.
- **Phase 3 — 일일 캡처 (검증 후).** 트렌드 외에 "오늘 내가 직접 배운 한 줄"을 빠르게 찍는 스킬. 실제로 필요한지 한 주쯤 운영해보고 결정.
- **Phase 4 — 인사이트 승격 (자연 발생).** summaries 여러 달치가 쌓이면 주제를 가로지르는 evergreen이 보인다. 그때 `notes/insights/`로 승격. 설계하지 않고 발견한다.

**5년 뒤 그림**: 60개 월간 요약, 20–30개 인사이트, 수십 편의 번역·노트. 얇은 개인 기술 메모어 한 권 분량. 그게 목표.

## 원칙

- **일일 습관과 인지 작업은 분리한다.** 매일 하는 동작은 얇게, 머리 쓰는 일은 월 1회 몰아서. 리추얼이 끊기는 순간 지식베이스는 죽는다.
- **회상은 검색이 아니다.** 사람은 날짜와 주제 덩어리로 기억한다. 파일을 쪼갤수록 Grep은 되지만 "그거 어디 있었지?"가 무너진다. 단일 raw 피드 + 월간 요약 랜드마크가 기억 구조와 맞는다.
- **AI에게 분류를 맡기지 않는다.** 수집까지는 맡겨도, 분류·증류·판단은 사람이 쥔다. 매일 codex의 태깅을 검증하기 시작하면 친밀감이 사라지고 일처럼 느껴진다.
- **죽은 디렉토리는 빨리 묻는다.** 과거의 실습 코드에 붙들려 지금의 용도를 희석시키지 않는다. `git log`가 보존해 주니까 마음 편히 삭제.
- **미리 쪼개지 않는다.** 프론트매터, 태그, 인덱스는 실제 불편이 생긴 뒤 도입. 지금 100개짜리 구조를 만들어도 10년 뒤엔 다시 짤 거다.
- **구조는 가설이다.** 이 README도 1년 뒤 다시 쓸 가능성이 높다. 그래도 지금 명시해 둬야 나중에 무엇이 틀렸는지 안다.

## 무엇이 아닌가

- **블로그 아님.** 독자를 향해 쓴 글이 아니다. 내가 나에게 남긴 메모 중 일부가 공개돼 있을 뿐.
- **Zettelkasten 아님.** 양방향 링크, 원자적 노트, 정교한 네트워크 — 그런 이상적 구조는 따라가지 않는다. 마크다운 + git + 읽을 수 있는 폴더명이면 충분.
- **지식 그래프 아님.** 관계보다 **문장**이 중요하다. 쿼리하고 싶은 게 아니라 읽고 싶은 거니까.
- **완성되는 것 아님.** 끝나는 날은 없다. 그저 더 두꺼워질 뿐.
